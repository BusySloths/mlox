## Ways to Contribute

Thank you for considering contributing to MLOX! Contributions are not limited to code:

- 🐛 Bug reports
- ✨ Enhancement suggestions
- 📚 Documentation improvements
- 🔁 Expanding reusable snippets and use-cases

MLOX is a community effort and still in an early stage — all ideas and improvements are welcome.

---

## Where to Start

1. **Browse** the [Issues tab](https://github.com/BusySloths/mlox/issues) — filter by `good-first-issue` for beginner-friendly tasks.
2. **Check** [GitHub Projects](https://github.com/BusySloths/mlox/projects) for current priorities.
3. **Ask** questions in [Discussions](https://github.com/BusySloths/mlox/discussions).
4. **Set up** your development environment (see [Installation](Installation)).

## Workflow Overview

### Project Hierarchy

```
🏗️ GitHub Projects (Strategic Areas)
    ├── 📊 Milestones (Releases)
    │   ├── 🎯 Issues (Tasks)
    │   │   └── 🔄 Pull Requests (Implementation)
    │   └── 🎯 Issues (Tasks)
    │       └── 🔄 Pull Requests (Implementation)
    └── 📊 Milestones (Releases)
        └── ...
```

### Current GitHub Projects

| Project | Description |
|---------|-------------|
| 🏗️ Infrastructure Core | Server management, networking |
| 🔧 Services & Integrations | MLOps services, Docker/K8s |
| 🎯 User Experience | CLI, Web UI, onboarding |
| 🔒 Security & Configuration | Auth, secrets, config |
| 📊 Monitoring & Observability | Logging, metrics, tracing |
| 🚀 Community & Growth | Docs, tutorials, community |

### Development Phase

1. Pick an issue
2. Create a branch
3. Develop the feature
4. Open a pull request
5. Review & merge

## Labels

### Type (Required)

| Label | Meaning |
|-------|---------|
| `type:bug` | Something broken |
| `type:feature` | New functionality |
| `type:enhancement` | Improvement to existing feature |
| `type:documentation` | Documentation changes |

### Priority

| Label | Meaning |
|-------|---------|
| `priority:critical` | Urgent, blocking |
| `priority:high` | Important |
| `priority:medium` | Standard |
| `priority:low` | Nice-to-have |

### Community

| Label | Meaning |
|-------|---------|
| `good-first-issue` | Beginner-friendly |
| `help-wanted` | Seeking community help |

## Issue & PR Checklists

### Creating an Issue

- [ ] Used appropriate issue template
- [ ] Clear, descriptive title with component prefix
- [ ] Applied type label (required)
- [ ] Applied component label
- [ ] Applied priority label
- [ ] Assigned to milestone (if applicable)
- [ ] Added to relevant project
- [ ] Included acceptance criteria (for features)
- [ ] Tagged relevant maintainers (if needed)

### Opening a Pull Request

- [ ] Used PR template
- [ ] Linked to related issues (`Fixes #123`)
- [ ] Added appropriate labels
- [ ] Requested review from relevant maintainers
- [ ] Ensured CI/CD passes
- [ ] Updated documentation (if needed)
- [ ] Added/updated tests (if applicable)
